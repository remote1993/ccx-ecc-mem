# Platform Integration Guide

This guide is for developers building Claude-Mem integrations such as IDE plugins, editor extensions, CLI tooling, and transcript-driven clients.

The current baseline is:
- heterogeneous host integrations
- a unified internal event model
- a local worker-backed runtime on `localhost`
- a custom third-party API extraction path
- compatibility MCP surfaces only where explicitly enabled

## Core Principle

Claude-Mem should be understood as:
- **externally heterogeneous** — each host integrates in its native way
- **internally unified** — normalized events flow into the same worker API and storage model

That means new platforms do **not** need to mimic Claude Code hooks if their native integration model is different.

## Integration Types

### Hook-based hosts

Use native lifecycle hooks when the host supports them.

Examples in the current repository include:
- Claude Code
- Cursor
- Gemini CLI
- Windsurf

Typical flow:

```text
host hook
  → worker-service.cjs hook <platform> <event>
  → platform adapter
  → normalized hook input
  → shared event handler
  → worker HTTP API
```

### Transcript-based hosts

Use transcript or session log ingestion when the host does not expose reliable hooks.

Current example:
- Codex CLI transcript watch integration

Typical flow:

```text
host transcript/session log
  → transcript watcher
  → parsed session/user/tool events
  → worker ingestion path
```

### Plugin-based hosts

Use the host's plugin runtime when that is the native extension surface.

Current examples:
- OpenCode
- OpenClaw

Typical flow:

```text
host plugin runtime
  → plugin-side capture/context sync
  → worker HTTP API
```

### MCP-based hosts

Use MCP only when the host primarily exposes Claude-Mem through MCP configuration.

Current examples include compatibility installers for:
- Copilot CLI
- Antigravity
- Goose
- Crush
- Roo Code
- Warp

This is a compatibility surface, not the primary runtime model.

## Unified Internal Contract

Regardless of host type, integrations should converge on the same internal concepts.

### Normalized event payload

The shared normalized hook shape is defined around:
- `sessionId`
- `cwd`
- `platform`
- `prompt`
- `toolName`
- `toolInput`
- `toolResponse`
- `transcriptPath`
- `metadata`
- optional subagent identity fields

See:
- `src/cli/types.ts:1`
- `src/cli/adapters/index.ts:1`

### Platform source normalization

Platform-specific names are reduced to a stable source label for storage and retrieval.

Examples:
- transcript-derived input → `codex`
- Gemini-family labels → `gemini`
- Claude-family labels → `claude`

See:
- `src/shared/platform-source.ts:1`

### Worker API contract

The worker remains the main internal contract for:
- session initialization
- observation intake
- summary generation
- session completion
- retrieval and search
- settings persistence

## Current Runtime Architecture

### End-to-end flow

```text
native host integration
  → normalized event or worker request
  → local worker runtime
  → session / search / settings / data routes
  → CustomApiAgent for extraction and summarization
  → SQLite / optional Chroma / optional MCP surfaces
```

### Why this boundary matters

This design allows the project to:
- support custom third-party APIs without tying to a specific host
- support multiple CLIs and tools without forcing one external integration pattern
- keep storage, search, and extraction logic centralized

## Integration Guidance

### Prefer native host capabilities first

When adding a new integration:
1. use the host's native extension model first
2. normalize into the shared internal contract
3. reuse the worker HTTP API and existing handlers
4. add MCP-specific behavior only when the host truly needs it

### Do not over-unify external installation/runtime behavior

Avoid assuming every host should:
- install through the same hook mechanism
- emit the same lifecycle vocabulary
- manage context through the same file
- expose the same permissions model

External behavior may differ. Internal processing should stay consistent.

### Preserve progressive disclosure

When surfacing historical context:
1. show compact search/index results first
2. let the caller narrow to relevant items
3. fetch full details only on demand

That matches the current retrieval design and reduces context waste.

## Troubleshooting Integrations

Start with the worker path:

```bash
curl http://localhost:37777/health
curl http://localhost:37777/api/readiness
curl http://localhost:37777/api/settings
npm run worker:logs
```

If requests fail, inspect:
- worker startup and readiness
- route wiring in the worker HTTP server
- local settings and custom API configuration
- the host-specific native integration layer

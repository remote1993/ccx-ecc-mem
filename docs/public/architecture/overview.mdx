# Architecture Overview

Claude-Mem now centers on a unified hook entry and a local worker-backed runtime. Lifecycle events enter through `plugin/hooks/hooks.json`, flow through the `src/cli/` handler pipeline, and hand off to the worker HTTP API for storage, retrieval, and asynchronous extraction.

## System Components

Claude-Mem operates as a memory plugin with six core components:

1. **Hook Entry Layer** - Capture lifecycle events through `plugin/hooks/hooks.json` → `worker-service.cjs hook claude-code <event>` → `src/cli/`
2. **Smart Install** - Cached dependency checker that runs before the main hook flow when needed
3. **Worker Service** - Orchestrates sessions, storage, search, settings, SSE, and custom API extraction over HTTP
4. **Database Layer** - Stores sessions and observations in SQLite with FTS5; Chroma remains optional for semantic retrieval
5. **Retrieval Surfaces** - Worker-backed HTTP retrieval is the primary path, with legacy MCP compatibility surfaces only where still enabled
6. **Viewer UI** - Web-based real-time memory stream visualization

## Primary Runtime Flow

### Hook-to-worker pipeline

```text
plugin/hooks/hooks.json
  → bun-runner.js / worker-service.cjs
  → worker-service.cjs hook claude-code <event>
  → src/cli/hook-command.ts
  → src/cli/handlers/*
  → local worker HTTP API
  → session runtime / search / settings routes
  → SQLite / optional Chroma
```

This is the current runtime shape. Older per-hook script names are historical references, not the primary model.

### Memory pipeline

```text
Hook stdin → src/cli handler → Worker HTTP API → Session manager / pending queue → Custom API extraction → SQLite / Chroma → Next session context
```

1. **Input**: Claude Code, Cursor, Gemini CLI, and other adapters send normalized hook data through the `src/cli/` pipeline.
2. **Ingest**: Hook handlers forward prompts and observations to the worker over localhost HTTP.
3. **Processing**: The worker queues pending messages and runs the custom API extraction path asynchronously.
4. **Output**: Structured observations and summaries are written back to SQLite and synced to Chroma when enabled.
5. **Retrieval**: Later hooks, search APIs, and the viewer read stored context from the worker.

### Retrieval pipeline

```text
User query → Worker-backed retrieval surface → Worker HTTP API → Search / data routes → SQLite FTS5 / optional Chroma → Formatted results
```

1. **User query**: A user asks for prior work, bugs, or decisions.
2. **Retrieval surface**: The current worker-backed path receives the request; legacy MCP surfaces can proxy to the same API if enabled.
3. **HTTP API**: Retrieval uses worker routes such as search, context, timeline, and batch observation reads.
4. **Storage/query layer**: SQLite FTS5 handles text search; optional Chroma supports semantic retrieval.
5. **Formatting**: The worker formats results for context injection, viewer rendering, or compatibility surfaces.

Uses progressive disclosure: search/index first, then timeline/context, then full observation details.

## Active Lifecycle Coverage

The current runtime covers these lifecycle events:

| Stage | Hook event | Main purpose |
|-------|------------|--------------|
| **0. Setup** | `Setup` | Ensure runtime dependencies are available |
| **1. SessionStart** | `SessionStart` | Start/health-check worker and inject prior context |
| **2. UserPromptSubmit** | `UserPromptSubmit` | Create/get session and save prompt |
| **3. PostToolUse** | `PostToolUse` | Queue observations for asynchronous processing |
| **4. PreToolUse (Read)** | `PreToolUse` | Inject file-context hints before `Read` |
| **5. Stop** | `Stop` | Summarize the session |
| **6. SessionEnd** | `SessionEnd` | Mark session complete |

This is broader than the older simplified 5-stage description.

## Component Details

### 1. Hook Entry Layer
- Built around `src/cli/hook-command.ts`, platform adapters, and lifecycle handlers
- Receives lifecycle events through the unified hook entry in `plugin/hooks/hooks.json`
- Normalizes inputs from Claude Code, Cursor, Gemini CLI, Windsurf, and raw integrations
- Hands off quickly to the worker HTTP API instead of doing heavy processing inline

**Note**: `src/hooks/` now contains only lightweight shared hook response helpers, not the main lifecycle implementations.

See [Hook Lifecycle](/architecture/hooks) for current hook-entry details.

### 2. Worker Service
The Bun-managed worker exposes the main localhost HTTP API and owns:
- session creation, observation intake, and summarization
- search, timeline, data, settings, and viewer routes
- readiness/health state and background initialization
- async custom API extraction and persistence
- SSE broadcasting for the viewer

See [Worker Service](/architecture/worker-service) for route structure and initialization flow.

### 3. Database Layer
SQLite with bun:sqlite driver provides:
- FTS5 virtual tables for full-text search
- session, prompt, observation, and summary persistence
- the primary durable store for context injection and retrieval
- optional Chroma synchronization for semantic retrieval

See [Database Architecture](/architecture/database) for schema and search details.

### 4. Search & Retrieval
Worker-backed retrieval provides:
- context injection for new sessions
- full-text search over prompts, observations, and summaries
- timeline-style inspection through HTTP endpoints
- optional semantic retrieval through Chroma
- compatibility MCP surfaces only as an add-on path where still enabled

See [Search Architecture](/architecture/search-architecture) for current retrieval flow and historical compatibility context.

### 5. Viewer UI
React + TypeScript web interface at http://localhost:37777 featuring:
- real-time memory stream via Server-Sent Events
- infinite scroll pagination with automatic deduplication
- project filtering and settings persistence
- self-contained HTML bundle (`viewer.html`)

Built with esbuild into a single-file deployment surface.
